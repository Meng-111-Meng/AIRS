# 框架对比实验 crispe qwen

## 提示词
```
Capacity and Role（技能和角色）
请你扮演一位资深需求工程师。你需要与学生、教师、教务处管理员进行虚拟访谈，并从复杂信息中精准识别出需求。

Insight（背景）
学校目前需要开发一个学校选课管理系统。

Statement（任务陈述）
请系统性地提取并整理"学校选课管理系统"的需求，包括：
识别所有关键利益相关者（如学生、教师、教务处管理员等）的核心诉求；

Personality（个性化）
回答需采用清晰、结构化、专业但易于理解的风格；

Experiment（实验）
最终将提取的需求转化为以系统为主语的需求描述
请为每条生成的需求标注类型（功能需求 / 非功能需求 / 约束性需求 / 质量需求），标注格式为【需求类型】：需求内容
生成完整版系统需求描述清单
```



## 结果
```
以下是以资深需求工程师身份，通过虚拟访谈学生、教师与教务处管理员后，系统性提取并整理的“学校选课管理系统”完整需求清单。所有需求均以系统为主语进行描述，并按【需求类型】分类标注，确保清晰、结构化且具备可追溯性。

一、关键利益相关者及其核心诉求概览

- 学生：希望便捷查询课程、公平选课、实时查看选课结果、避免冲突、及时获得通知。
- 教师：需发布课程信息、查看选课名单、管理教学班容量、调整课程状态。
- 教务处管理员：需统筹课程计划、控制选课流程（如开启/关闭选课窗口）、处理异常情况、生成统计报表、保障系统稳定与数据安全。

二、系统需求描述清单

功能需求（Functional Requirements）

【功能需求】：系统应允许学生在指定选课时间段内浏览所有开放课程的详细信息（包括课程名称、课程代码、授课教师、学分、上课时间、地点、已选人数、剩余容量等）。

【功能需求】：系统应支持学生根据课程类别、教师姓名、上课时间等条件对课程列表进行筛选和排序。

【功能需求】：系统应允许学生将感兴趣的课程加入“待选清单”，并在正式选课时快速提交。

【功能需求】：系统应在学生提交选课请求时，自动检查是否存在时间冲突、先修课程未完成、学分上限超限等情况，并阻止无效选课操作。

【功能需求】：系统应支持学生在选课开放期间修改或退选已选课程，并实时更新课程容量状态。

【功能需求】：系统应向学生实时显示其当前已选课程列表、总学分及是否满足培养方案要求（若集成培养方案模块）。

【功能需求】：系统应允许教师创建、编辑和发布其所授课程的基本信息（仅限本人课程），并设置最大选课人数。

【功能需求】：系统应在选课结束后自动生成每门课程的学生名单，并供授课教师下载或在线查看。

【功能需求】：系统应允许教师在选课截止前临时关闭或重新开放某门课程的选课权限（需经教务审核或具备权限控制）。

【功能需求】：系统应支持教务处管理员批量导入/导出课程数据（如通过Excel模板），并维护全校课程目录。

【功能需求】：系统应提供选课阶段管理功能，允许教务处管理员设定不同轮次的选课开始与结束时间（如预选、正选、补退选）。

【功能需求】：系统应在选课高峰期支持高并发访问，并采用排队或抽签机制处理超额选课请求（如热门课程）。

【功能需求】：系统应记录所有用户的选课操作日志（包括时间、操作类型、课程ID、用户ID），供审计与问题追溯。

【功能需求】：系统应支持教务处管理员手动干预个别学生的选课结果（如因特殊原因强制加课或退课）。

【功能需求】：系统应定期自动生成选课统计报表（如各课程选课人数分布、专业选课完成率、教师开课负载等），并支持导出为PDF或Excel格式。

非功能需求（Non-functional Requirements）

【非功能需求】：系统应在正常网络条件下，95%以上的页面响应时间不超过2秒。

【非功能需求】：系统应支持至少3000名学生同时在线选课而不出现服务中断或严重延迟。

【非功能需求】：系统应提供7×24小时可用性，除计划内维护窗口外，年可用率不低于99.5%。

【非功能需求】：系统界面应符合WCAG 2.1 AA级无障碍标准，确保视障或行动不便用户可正常使用。

【非功能需求】：系统应支持主流浏览器（Chrome、Edge、Firefox、Safari）及移动端适配。

质量需求（Quality Requirements）

【质量需求】：系统必须保证选课数据的一致性与完整性，任何选课操作不得导致课程容量负值或重复选课。

【质量需求】：系统应具备容错能力，在网络中断或服务器短暂故障后能恢复未完成的事务，避免数据丢失。

【质量需求】：系统应提供清晰的用户反馈（如成功提示、错误说明、操作指引），降低用户误操作率。

约束性需求（Constraints）

【约束性需求】：系统必须与学校现有的统一身份认证平台（如LDAP或CAS）集成，实现单点登录。

【约束性需求】：系统开发与部署必须符合《中华人民共和国网络安全法》及教育行业数据安全规范，学生个人信息不得明文存储。

【约束性需求】：系统数据库设计需兼容学校现有教务数据库结构（如Oracle或MySQL），并支持未来与学业预警、毕业审核等系统对接。

【约束性需求】：系统前端界面语言默认为简体中文，但需预留多语言扩展接口。

三、总结说明

以上需求清单基于典型高校场景提炼，实际项目中仍需结合具体校情（如学分制类型、选课规则复杂度、信息化基础）进一步细化。建议后续开展用例建模、原型验证与优先级排序（如MoSCoW法），以指导迭代开发。

如需进一步拆解为用户故事、绘制用例图或定义验收标准，可继续提出。
```

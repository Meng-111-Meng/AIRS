# 框架对比实验 roses qwen

## 提示词
```
角色 (Role):
请你作为一名资深的需求工程师。

目标 (Objective):
你需要与学生、教师、教务处管理员进行虚拟访谈，并从复杂信息中精准识别出需求。

场景 (Scenario):
学校目前需要开发一个学校选课管理系统。

预期解决方案 (Expected Solution):
将提取的需求转化为以系统为主语的需求描述，最终生成完整版系统需求描述清单

步骤 (Steps):
1. 识别所有关键利益相关者的核心诉求；
2. 将核心诉求转化为以系统为主语、明确具体、可验证的需求描述；
3. 请为每条生成的需求标注类型（功能需求 / 非功能需求 / 约束性需求 / 质量需求），标注格式为【需求类型】：需求内容。
4. 生成完整版系统需求描述清单。
```


## 结果
```
作为资深需求工程师，我将通过虚拟访谈方式，从学生、教师和教务处管理员三类关键利益相关者中提取核心诉求，并按照规范步骤转化为系统需求描述。以下是完整的需求分析与转化过程：

第一步：识别关键利益相关者的核心诉求

1. 学生视角
- 希望能清晰查看可选课程（含课程简介、教师、时间、容量等）；
- 能在规定时间内自主选课、退课；
- 选课结果实时反馈，避免“抢不到课”却无提示；
- 查看已选课程、学分统计及冲突检测；
- 选课期间系统响应快，不卡顿。

2. 教师视角
- 能发布自己开设的课程信息（包括课程大纲、先修要求、最大人数等）；
- 可查看选课学生名单并导出；
- 能设置课程选课条件（如专业限制、年级限制）；
- 在选课结束后可调整学生名单（如补录、退选审核）；
- 系统操作简洁，界面友好。

3. 教务处管理员视角
- 能统一管理全校课程池（增删改查课程基本信息）；
- 设置选课轮次、开放/关闭时间；
- 监控选课整体情况（如热门课程、未达开班人数课程）；
- 处理异常情况（如学生特殊选课申请、课程冲突仲裁）；
- 系统需具备高可靠性、数据安全性和审计日志；
- 支持与现有教务系统（如学籍、成绩系统）集成。

第二至第四步：转化为系统需求描述清单

以下为以系统为主语、明确具体、可验证的需求描述，并标注需求类型：

完整版系统需求描述清单

【功能需求】：系统应允许学生在规定的选课时间段内浏览所有开放课程的详细信息，包括课程名称、课程代码、授课教师、上课时间、地点、学分、课程简介、先修要求、当前选课人数与容量上限。

【功能需求】：系统应支持学生执行选课和退课操作，并在操作完成后立即返回成功或失败的状态提示（如“选课成功”、“课程已满”、“时间冲突”等）。

【功能需求】：系统应在学生提交选课请求时，自动检测课程时间冲突、先修课程未完成、专业/年级限制不符等条件，并阻止不符合条件的选课操作。

【功能需求】：系统应为学生提供个人课表视图，实时展示已选课程的时间安排，并自动计算当前学期总学分。

【功能需求】：系统应允许教师创建、编辑其负责开设的课程信息，包括设置最大选课人数、选课限制条件（如仅限某专业、某年级）及课程描述。

【功能需求】：系统应在选课结束后，向教师提供其所授课程的最终学生名单，并支持以 CSV 或 Excel 格式导出。

【功能需求】：系统应支持教师在特定权限窗口期内，对课程学生名单进行手动调整（如批准补选、移除学生），并记录操作日志。

【功能需求】：系统应允许教务处管理员创建、修改、停用或删除全校范围内的课程记录，并指定课程所属院系、学期、类型（必修/选修）等元数据。

【功能需求】：系统应支持教务处管理员配置多轮选课计划，包括每轮的开始时间、结束时间、适用对象（如高年级优先）及是否允许退课。

【功能需求】：系统应提供实时选课监控仪表盘，展示各课程的选课人数、满课率、未达最低开班人数的课程列表等统计信息。

【功能需求】：系统应支持处理特殊选课申请流程，允许学生提交加课申请，教师审核，教务处备案，并在审核通过后自动加入课表。

【功能需求】：系统应提供完整的操作审计日志，记录关键操作（如选课、退课、课程调整、管理员变更）的操作人、时间、操作内容及IP地址。

【非功能需求】：系统在选课高峰期（如首轮开放前30分钟）应支持不少于5000名用户并发操作，平均响应时间不超过2秒。

【非功能需求】：系统应保证99.9%的可用性，在非维护时段不得无故中断服务。

【质量需求】：系统界面应符合无障碍设计标准，支持主流浏览器（Chrome、Edge、Firefox 最新两个版本）及移动端适配。

【质量需求】：系统应对所有用户输入进行合法性校验，防止SQL注入、XSS等常见Web攻击。

【约束性需求】：系统必须与学校现有的统一身份认证系统（如LDAP或OAuth2）集成，实现单点登录。

【约束性需求】：系统数据存储须符合《教育信息系统安全等级保护基本要求》（等保二级或以上），敏感数据（如学生信息）须加密存储。

【功能需求】：系统应在每学期选课结束后，自动生成结构化数据接口，供成绩录入系统和学籍管理系统调用课程-学生关联数据。

【非功能需求】：系统应提供操作引导和帮助文档，新用户可在5分钟内完成首次选课操作。

该需求清单覆盖了核心业务流程、用户体验、安全性、性能与合规性，可作为后续系统设计、开发与验收的基准依据。
```

# 学校选课管理系统需求规格说明书（精简版）

## 一、关键利益相关者及其核心诉求

| 角色 | 核心诉求 |
|------|----------|
| **学生** | 1. 便捷地查询可选课程信息（时间、教师、名额等）<br>2. 公平、流畅的选课/退课操作体验<br>3. 实时查看个人课表与选课结果<br>4. 系统稳定，尤其在选课高峰期间 |
| **教师** | 1. 查询自己所授课程的选课学生名单<br>2. 提交课程成绩并管理<br>3. 获知课程选课人数与调整情况<br>4. 维护个人课程基本信息（如大纲、考核方式） |
| **教务处管理员** | 1. 全面管理课程库与开课计划<br>2. 设定选课规则与时间安排<br>3. 监控选课过程，处理异常与冲突<br>4. 生成各类统计报表（选课率、班级规模等）<br>5. 保障系统数据安全与操作审计 |

## 二、系统需求描述清单

### 1. 课程信息管理
- **【功能需求】**：系统应允许教务处管理员创建、修改、发布和归档课程信息，包括课程代码、名称、学分、学时、授课教师、上课时间地点、容量限制、课程简介及先修课程要求。
- **【功能需求】**：系统应允许教师对其所授课程的教学大纲、考核方式等详细信息进行维护。
- **【功能需求】**：系统应为学生和教师提供多条件（如课程名、教师、时间）组合查询课程信息的界面。

### 2. 选课流程与规则
- **【功能需求】**：系统应依据教务处设置的选课阶段（如预选、正选、补退选）和时间窗口，控制选课功能的开放与关闭。
- **【功能需求】**：系统应在学生进行选课操作时，自动校验其是否满足先修课程要求、时间是否冲突、课程容量是否已满，并即时给出明确提示。
- **【功能需求】**：系统应支持学生在规定时间内进行退课操作，并实时释放相应课程名额。
- **【约束性需求】**：系统必须遵循学校制定的选课优先规则（如高年级优先、本专业优先等）并在选课过程中自动应用。

### 3. 课表与名单查询
- **【功能需求】**：系统应为学生生成并展示个人实时课表（包括课程名称、时间、地点、教师）。
- **【功能需求】**：系统应允许教师按课程查询并导出选课学生名单（含学号、姓名、班级等信息）。
- **【功能需求】**：系统应为教务处管理员提供按课程、班级、教师等多维度查看选课情况的功能。

### 4. 成绩管理
- **【功能需求】**：系统应在成绩录入期，允许授课教师为所选课程的学生录入、修改和提交成绩。
- **【功能需求】**：系统应确保成绩提交后，需经教务处管理员审核或确认后方正式生效。
- **【功能需求】**：系统应允许学生在成绩发布后查询自己的各科成绩。

### 5. 统计与报表
- **【功能需求】**：系统应能自动生成选课人数统计、课程选课率、班级规模分布等报表，供教务处管理员查看和导出。
- **【功能需求】**：系统应提供选课操作日志查询功能，记录关键操作（如选课、退课、成绩录入）的人员、时间和内容。

### 6. 系统性能与质量
- **【非功能需求 / 性能需求】**：系统在选课高峰时段（如正选开放首小时）应能支持至少5000名用户并发进行选课相关操作，页面平均响应时间不超过3秒。
- **【质量需求 / 可靠性需求】**：系统全年计划内可用性应不低于99.5%，关键业务数据（选课记录、成绩）必须每日备份。
- **【质量需求 / 安全性需求】**：系统必须实现基于角色的访问控制（RBAC），确保用户只能访问其权限范围内的功能和数据。所有数据传输需采用加密协议（如HTTPS）。
- **【质量需求 / 易用性需求】**：系统用户界面应直观清晰，对于主要业务流程（如选课、查课表），未经培训的用户应在5分钟内完成操作。

### 7. 系统约束
- **【约束性需求】**：系统必须与学校现有的统一身份认证平台对接，实现单点登录。
- **【约束性需求】**：系统后端数据库必须采用学校指定的MySQL 8.0或以上版本。
- **【约束性需求】**：系统开发需遵循学校信息中心提供的API接口规范和数据标准。

## 三、总结

本需求清单通过虚拟访谈，系统性地提取并转化了学生、教师、教务处管理员三类核心用户的核心诉求，形成了以**“系统应……”**为主语的结构化需求描述。清单涵盖了功能、非功能（性能、质量）及约束性需求，为后续的选课管理系统设计、开发与测试提供了明确的依据和范围。在项目启动前，建议与各方利益相关者对上述需求进行正式评审与确认。

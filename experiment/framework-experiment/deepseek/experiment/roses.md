# 框架对比实验 roses deepseek

## 提示词
```
角色 (Role):
请你作为一名资深的需求工程师。

目标 (Objective):
你需要与学生、教师、教务处管理员进行虚拟访谈，并从复杂信息中精准识别出需求。

场景 (Scenario):
学校目前需要开发一个学校选课管理系统。

预期解决方案 (Expected Solution):
将提取的需求转化为以系统为主语的需求描述，最终生成完整版系统需求描述清单

步骤 (Steps):
1. 识别所有关键利益相关者的核心诉求；
2. 将核心诉求转化为以系统为主语、明确具体、可验证的需求描述；
3. 请为每条生成的需求标注类型（功能需求 / 非功能需求 / 约束性需求 / 质量需求），标注格式为【需求类型】：需求内容。
4. 生成完整版系统需求描述清单。
```




## 结果
```
### **学校选课管理系统需求描述清单**

**1.0 需求概述**
本需求清单旨在定义学校选课管理系统的核心功能与非功能要求。系统需服务于学生、教师及教务处管理员三类核心用户，实现课程发布、选课、管理与统计分析的全流程数字化，提升选课效率与公平性。

**2.0 利益相关者核心诉求与转化需求**

**2.1 学生用户**
*   **核心诉求**：便捷、公平地完成选课、退课操作，查询个人课表与可选课程。
*   **转化需求**：
    1.  【功能需求】：系统应允许学生根据个人培养方案、学期、时间等条件查询可选课程列表，并查看课程的详细信息（如教师、学分、时间地点、已选人数/容量）。
    2.  【功能需求】：系统应允许学生在规定的选课时间段内，对符合条件的课程进行选课与退课操作。
    3.  【功能需求】：系统在学生选课时，应能自动检测并提示与已选课程的时间冲突。
    4.  【功能需求】：系统应提供个人课表（可视化日历视图）与已选课程列表的查看功能。
    5.  【功能需求】：系统应支持多轮次选课（如正选、补退选），并为热门课程提供公平的抽签或先到先得机制。
    6.  【非功能需求】：系统在选课高峰期（如选课开放瞬间）需支持高并发访问，页面响应时间不应超过3秒。

**2.2 教师用户**
*   **核心诉求**：发布和管理个人课程，查看选课学生名单，并可能进行初步筛选或成绩关联。
*   **转化需求**：
    7.  【功能需求】：系统应允许教师在规定时间内提交开课申请，填写课程信息（名称、大纲、容量、时间要求等）。
    8.  【功能需求】：系统应允许教师查询本人所授课程的选课学生名单，并支持导出。
    9.  【功能需求】：系统应允许教师对需要筛选学生的课程，在系统内提交初步的学生筛选名单建议（最终由教务处审核确认）。
    10. 【约束性需求】：教师只能管理与自己工号绑定的课程信息。
    11. 【质量需求】：系统向教师展示学生名单的界面应清晰、直观，支持按学号、姓名等关键信息排序与查找。

**2.3 教务处管理员**
*   **核心诉求**：全局管理课程库、选课规则与周期，监控选课进程，处理异常，并生成各类统计报表。
*   **转化需求**：
    12. 【功能需求】：系统应提供后台管理界面，供管理员对全校课程库进行增、删、改、查及状态（如开放选课/关闭）管理。
    13. 【功能需求】：系统应允许管理员灵活配置选课参数，包括选课轮次、各轮次起止时间、面向年级/专业等。
    14. 【功能需求】：系统应具备选课进程监控面板，实时显示总选课人次、热门课程饱和度等关键指标。
    15. 【功能需求】：系统应提供手动处理特定异常选课申请（如跨专业选课、学分超限特批）的通道。
    16. 【功能需求】：系统应能根据管理员的查询条件（如按学院、专业、课程、教师），生成选课情况统计报表，并支持导出为常见格式（如Excel, PDF）。
    17. 【非功能需求】：系统所有核心数据操作（如课程调整、选课结果确认）必须记录详细的操作日志（操作人、时间、内容），以备审计。
    18. 【约束性需求】：系统必须与学校现有统一身份认证系统集成，用户使用统一账号密码登录。
    19. 【约束性需求】：系统数据需与学生学籍管理系统、成绩管理系统实现数据接口对接，保证基础数据同步。

**3.0 全局性需求**
*   20. 【非功能需求/安全】：系统必须保证数据传输与存储的安全，防止信息泄露。学生个人成绩、教师评价等敏感信息需有严格的权限控制。
*   21. 【非功能需求/可靠性】：系统在选课期间应保证7x24小时高可用，计划外停机时间每年不得超过8小时。
*   22. 【质量需求/易用性】：系统的用户界面（UI）应简洁、一致，符合主流操作习惯，确保三类用户经过简单指导即可独立完成核心操作。
*   23. 【约束性需求】：系统开发需遵循学校信息化部门制定的技术架构与数据标准。

**4.0 完整版系统需求描述清单**
1.  【FR】系统应允许学生根据多种条件查询可选课程及其详情。
2.  【FR】系统应允许学生在规定时间内进行选课与退课。
3.  【FR】系统应能在学生选课时自动检测时间冲突并给出提示。
4.  【FR】系统应提供个人课表与已选课程列表的查看功能。
5.  【FR】系统应支持多轮次选课机制，并为热门课程提供抽签或排队功能。
6.  【NFR】系统在选课高峰期需支持高并发，页面响应时间≤3秒。
7.  【FR】系统应允许教师提交开课申请并管理课程信息。
8.  【FR】系统应允许教师查看并导出所授课程的学生名单。
9.  【FR】系统应允许教师提交课程的学生筛选建议名单。
10. 【CR】系统应确保课程数据操作权限与教师工号严格绑定。
11. 【QR】系统向教师提供的名单管理界面应清晰、易操作。
12. 【FR】系统应提供后台管理界面，供管理员全面管理课程库。
13. 【FR】系统应允许管理员配置选课规则与时间参数。
14. 【FR】系统应具备实时选课进程监控功能。
15. 【FR】系统应提供处理异常选课申请的人工通道。
16. 【FR】系统应能生成并导出多样化的选课统计报表。
17. 【NFR】系统必须记录所有核心数据操作的完整审计日志。
18. 【CR】系统必须与学校统一身份认证系统集成。
19. 【CR】系统必须与学籍、成绩管理系统实现数据接口对接。
20. 【NFR】系统必须具备安全保障机制，对敏感数据进行权限控制。
21. 【NFR】系统在选课期间应保证高可用性（可用性目标>99.9%）。
22. 【QR】系统用户界面应简洁、一致、易用。
23. 【CR】系统开发需符合学校既定技术架构与数据标准。

**【注】需求类型缩写：**
*   **FR**：功能需求
*   **NFR**：非功能需求
*   **QR**：质量需求
*   **CR**：约束性需求

此清单可作为系统设计、开发与测试的基准依据。在后续详细设计阶段，每项需求需进一步分解为更具体的技术任务与验收标准。
```

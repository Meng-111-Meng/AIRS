# 纵向实验(AIRS) Qwen3-Max

## 提示词
```
1. 锚定 
角色：资深需求工程师
背景：一所综合性大学计划开发新的“学校选课管理系统”。项目处于最早期阶段。
任务：目前在锚定阶段，请你先明确所需扮演的角色，背景和任务。
在接下来的推演阶段，你将执行初始的需求获取工作。具体而言，当进入推演阶段时，你需要模拟一次与关键干系人（学生、教师、教务处管理员）的引导式访谈，通过提问来发掘和记录他们的原始问题、操作流程、痛点及期望，并分类成功能需求、非功能需求、约束性需求、质量需求，最终输出一份需求描述清单。
如果已明确请提示可进入【推演】阶段



2. 推演
现在，基于以上锚定，开始需求探索。
请你模拟对“本科生”、“任课教师”和“教务管理员”这三类关键用户进行访谈。请分别列出从每类用户那里搜集到的原始反馈。反馈内容应聚焦于他们在实际操作中遇到的具体问题、不便之处、以及他们直接表达的期望。请使用他们可能使用的自然语言来描述，确保反馈是具体、有场景的，而非笼统的概括。

请严格按以下格式输出：
一、 本科生主要反馈：
[问题描述]：例如，“选课系统在早上9点开放时，经常点不动或者直接报错，等能进去的时候，热门的通选课早就没名额了。”
[具体场景与期望]：例如，“我在安排课程表时，需要自己用纸笔记下时间，再手动比对有没有冲突。希望系统能在我尝试选择有冲突的课时，立刻用红色高亮提示我具体跟哪门课冲突了。”
...（继续列出其他反馈）

二、 任课教师主要反馈：
[问题描述]：例如，“课程结束后，我需要从好几个不同的Excel表格里手动汇总、核对选课学生名单和最终成绩单，非常容易出错，也很耗时。”
[具体场景与期望]：例如，“我希望在课程容量满员后，系统能自动关闭选课通道，而不是需要我每天登陆上去手动查看并关闭。”
...（继续列出其他反馈）

三、 教务管理员主要反馈：
[问题描述]：例如，“目前每学期初都需要人工审核大量的课程替代、学分认定申请，这些申请格式不一，通过邮件或纸质提交，整理和审批流程混乱。”
[具体场景与期望]：例如，“我们需要系统能根据教学计划，自动生成不同专业、不同年级学生的推荐选课列表，作为他们的参考，这样可以减少大量无效的咨询工作。”
...（继续列出其他反馈）



3. 检验
现在对已收集到的原始诉求进行检验。
请从以下两个视角进行评估：
1.  清晰度视角：找出所有表述模糊、存在歧义或过于宽泛的诉求（例如，“系统要好用”、“操作要快”），并将其标记出来。
2.  完整性视角：基于我们已识别的用户角色和核心意图，你认为还有哪些重要的用户角色（如系统运维人员、财务处等）或业务场景（如选修课退补选、跨专业选课）的诉求可能被遗漏了？
3. 需求类型覆盖度视角：检查是否包含功能、非功能、约束性、质量四类需求，若存在类型缺失，需标记为遗漏项。

请直接列出你发现的具体模糊项和可能遗漏的视角以及用户角色。
若没有发现具体模糊项或可能遗漏的视角，且四类需求均覆盖完整，则提示“可以进入交付阶段”

4. 推进

基于上述检验生成后的结果，我们决定进行【优化 】。

若您决定进行【优化】，请继续：
作为需求工程师，请你根据检验结果指出的具体问题，执行对应操作：
A. 若检验指出存在模糊/歧义诉求：
请直接修改这些模糊诉求。将其拆解或转化为具体、无歧义、可作为后续设计输入的清晰陈述。
B. 若检验指出存在遗漏的视角/场景：
请模拟对已识别的遗漏角色或场景进行一次快速需求发现，并新增从该视角获取的原始诉求。

完成上述操作后，请按顺序提供以下交付物：
更新后的原始需求清单：整合所有优化后的诉求，按角色分类清晰列出。



基于上述检验生成后的结果，我们决定进行【交付 】。
请将最终获取的所有原始需求，按用户角色分类，整理成一份《系统需求描述清单》。
请为每条生成的需求标注类型（功能需求 / 非功能需求 / 约束性需求 / 质量需求），标注格式为【需求类型】：需求内容。
清单应清晰列出每个角色及其诉求，诉求需以系统为主语，明确具体、可验证。
最终只需输出《系统需求描述清单》即可
```



## 结果
```
《系统需求描述清单》

一、本科生相关需求  
【功能需求】：系统应在选课开放瞬间支持高并发访问，确保用户能正常提交选课请求，避免因系统卡顿或崩溃导致热门课程名额被不公平抢占。  
【功能需求】：系统在用户尝试选择时间冲突的课程时，应立即以红色高亮方式提示冲突的具体课程名称、上课时间和地点。  
【功能需求】：系统应提供可视化课表预览功能，允许学生在正式提交前拖拽课程模块进行排课模拟，并自动检测时间、教室或教师冲突。  
【非功能需求】：系统响应时间在正常网络条件下（如校园Wi-Fi）应不超过2秒，高峰期（如选课首日）不超过5秒。  
【功能需求】：系统应支持学生查看每门课程的实时余量、已选人数、教师评价摘要及课程大纲，辅助选课决策。  
【质量需求】：系统界面应符合无障碍设计标准，支持屏幕阅读器，并对色盲用户友好（如不依赖颜色传递关键信息）。  

二、任课教师相关需求  
【功能需求】：系统应在课程达到设定容量上限后自动关闭该教学班的选课入口，并向教师发送通知。  
【功能需求】：系统应自动生成并维护本课程的最终选课学生名单（含学号、姓名、专业、年级），并与成绩录入模块无缝对接，避免手动导入导出。  
【功能需求】：系统应支持教师在课程开始前设置先修课程要求、专业限制或年级限制，不符合条件的学生无法选入。  
【功能需求】：系统应提供课程调停课申请入口，教师可在线提交调课请求，自动通知已选学生并同步更新课表。  
【非功能需求】：成绩录入界面加载速度应低于1秒，且支持批量导入/导出（如Excel格式），操作失败时需明确提示错误行及原因。  

三、教务管理员相关需求  
【功能需求】：系统应根据各专业培养方案，自动生成分年级、分学期的推荐课程列表，并在学生选课界面显著位置展示。  
【功能需求】：系统应提供统一的线上通道处理课程替代与学分认定申请，支持上传证明材料、流程跟踪和电子审批留痕。  
【功能需求】：系统应支持按学院、专业、年级等维度批量设置课程属性（如必修/选修、学分、开课学期），并校验教学计划一致性。  
【约束性需求】：系统必须部署在校内私有服务器或通过等保三级认证的教育云平台，不得使用境外SaaS服务。  
【非功能需求】：系统需支持每学期至少5000门课程、10万学生同时在线选课，峰值并发用户不低于2万人。  
【质量需求】：系统全年可用性不低于99.5%，重大故障恢复时间不超过30分钟，并具备完整操作日志审计能力。  

四、补充角色：系统运维人员（基于完整性视角新增）  
【非功能需求】：系统应提供实时监控面板，显示CPU、内存、数据库连接数、API响应延迟等关键指标，并支持阈值告警。  
【约束性需求】：系统必须兼容学校现有统一身份认证（如LDAP/ OAuth 2.0）和数据中心接口规范。  
【功能需求】：系统应支持一键回滚至最近一次稳定选课状态，用于应对大规模数据异常或误操作。  

五、补充场景：跨专业/跨校选课（基于完整性视角新增）  
【功能需求】：系统应允许符合条件的学生申请跨专业或跨校（如联盟高校）选课，自动校验学籍状态、学分上限及对方课程开放权限。  
【功能需求】：跨校选课结果应自动同步至本校成绩单，并标注来源院校及课程性质，用于毕业审核。  

—— 清单结束 ——
```

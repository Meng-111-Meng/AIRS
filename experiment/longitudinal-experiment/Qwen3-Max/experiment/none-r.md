## 推演
```
现在，基于以上锚定，开始需求探索。
请你模拟对“本科生”、“任课教师”和“教务管理员”这三类关键用户进行访谈。请分别列出从每类用户那里搜集到的原始反馈。反馈内容应聚焦于他们在实际操作中遇到的具体问题、不便之处、以及他们直接表达的期望。请使用他们可能使用的自然语言来描述，确保反馈是具体、有场景的，而非笼统的概括。

请严格按以下格式输出：
一、 本科生主要反馈：
[问题描述]：例如，“选课系统在早上9点开放时，经常点不动或者直接报错，等能进去的时候，热门的通选课早就没名额了。”
[具体场景与期望]：例如，“我在安排课程表时，需要自己用纸笔记下时间，再手动比对有没有冲突。希望系统能在我尝试选择有冲突的课时，立刻用红色高亮提示我具体跟哪门课冲突了。”
...（继续列出其他反馈）

二、 任课教师主要反馈：
[问题描述]：例如，“课程结束后，我需要从好几个不同的Excel表格里手动汇总、核对选课学生名单和最终成绩单，非常容易出错，也很耗时。”
[具体场景与期望]：例如，“我希望在课程容量满员后，系统能自动关闭选课通道，而不是需要我每天登陆上去手动查看并关闭。”
...（继续列出其他反馈）

三、 教务管理员主要反馈：
[问题描述]：例如，“目前每学期初都需要人工审核大量的课程替代、学分认定申请，这些申请格式不一，通过邮件或纸质提交，整理和审批流程混乱。”
[具体场景与期望]：例如，“我们需要系统能根据教学计划，自动生成不同专业、不同年级学生的推荐选课列表，作为他们的参考，这样可以减少大量无效的咨询工作。”
...（继续列出其他反馈）
```

**结果**
```
一、 本科生主要反馈：  
[问题描述]：选课系统在早上9点开放时，经常点不动或者直接报错，等能进去的时候，热门的通选课早就没名额了。  
[具体场景与期望]：我在安排课程表时，需要自己用纸笔记下时间，再手动比对有没有冲突。希望系统能在我尝试选择有冲突的课时，立刻用红色高亮提示我具体跟哪门课冲突了。  

[问题描述]：有时候看到一门课还有余量，但点“选课”后却提示“人数已满”，刷新一下又显示有空位，反复几次就错过了。  
[具体场景与期望]：能不能在选课页面实时显示当前排队人数和预计等待时间？或者至少让我知道是不是真的还能选上。  

[问题描述]：退课之后，系统不会自动推荐其他时间合适、学分匹配的替代课程，只能自己一页页翻。  
[具体场景与期望]：如果我退掉一门周二下午的课，系统能不能自动给我列出所有周二下午可选、且符合我专业要求的其他课程？  

[问题描述]：课程简介太简略了，只有课程名和教师名，根本不知道这门课到底讲什么、作业多不多、有没有期中考试。  
[具体场景与期望]：希望每门课下面能展示往届学生的匿名评价、教学大纲摘要，甚至教师上传的课程说明视频。  

二、 任课教师主要反馈：  
[问题描述]：课程结束后，我需要从好几个不同的Excel表格里手动汇总、核对选课学生名单和最终成绩单，非常容易出错，也很耗时。  
[具体场景与期望]：我希望系统能在一个界面里直接导出最终确认的选课名单（含学号、姓名、学院），并支持我在线录入平时分、期中、期末成绩，自动计算总评。  

[问题描述]：有些学生明明没来上课，但因为没及时退课，期末还得给他们打分，甚至影响课程及格率统计。  
[具体场景与期望]：能不能在开学前两周设置一个“试听期”，系统自动标记未签到或未提交首次作业的学生，并提醒他们是否要正式选课？  

[问题描述]：我想限制选课学生的先修课程条件（比如必须修过“高等数学”才能选“概率论”），但现在的系统没法设置这种规则。  
[具体场景与期望]：希望在发布课程时，能勾选必要的先修课程或专业门槛，系统自动拦截不符合条件的学生选课申请。  

[问题描述]：课程容量满了之后，还有学生不断发邮件求加课，我得一个个手动记录，再统一找教务加白名单。  
[具体场景与期望]：能不能让学生在满员时直接加入“候补队列”？一旦有人退课，系统按顺序自动补录，并邮件通知。  

三、 教务管理员主要反馈：  
[问题描述]：目前每学期初都需要人工审核大量的课程替代、学分认定申请，这些申请格式不一，通过邮件或纸质提交，整理和审批流程混乱。  
[具体场景与期望]：我们需要一个统一的线上申请入口，学生上传材料后，系统自动关联其培养方案，标红不符合常规路径的部分，供我们重点审核。  

[问题描述]：排课时经常出现教室资源冲突，比如两个学院同时预订了同一间教室的同一时段，直到开课前一周才发现。  
[具体场景与期望]：希望系统能集成全校教室资源日历，在教师提交课程时间时，实时校验教室是否已被占用，并给出可用时间段建议。  

[问题描述]：每学期末要手动统计各专业选课完成率、课程负荷分布、不及格率等数据，用于教学评估，过程繁琐且易错。  
[具体场景与期望]：系统应能自动生成多维度的教学运行报表，支持按学院、专业、年级、课程类型等条件筛选和导出。  

[问题描述]：跨校区选课的学生经常搞错上课地点，跑到错误的校区，耽误一整天。  
[具体场景与期望]：在选课结果确认页和课表页面，必须清晰标注每门课的校区、楼栋、教室，并支持一键同步到手机日历（带地图链接）。
```



## 推进

### 交付
```
若您决定进行【交付】，请继续：
请将最终获取的所有原始需求，按用户角色分类，整理成一份《系统需求描述清单》。
请为每条生成的需求标注类型（功能需求 / 非功能需求 / 约束性需求 / 质量需求），标注格式为【需求类型】：需求内容。
清单应清晰列出每个角色及其诉求，诉求需以系统为主语，明确具体、可验证。
```

**结果**
```
【交付】阶段启动

《学校选课管理系统需求描述清单》

一、本科生相关需求  
1. 【功能需求】：系统应在选课开放时段内支持高并发访问，确保用户可正常提交选课请求。  
2. 【功能需求】：系统在用户尝试选择时间冲突课程时，应提示冲突的具体课程信息。  
3. 【功能需求】：系统应显示每门课程的当前剩余名额与排队人数，并在用户点击"选课"后提供明确的反馈。  
4. 【功能需求】：当用户退选某门课程后，系统应根据课程特征和用户培养方案，推荐符合条件的替代课程。  
5. 【功能需求】：系统应为每门课程提供详细信息页面，包括教学大纲摘要、教师说明材料、学生评价等。  
6. 【非功能需求】：系统在选课高峰期的页面响应时间应较短，且成功率较高。  
7. 【质量需求】：系统界面应标注每门课程的校区和教室信息，并支持将个人课表导出至日历应用。  

二、任课教师相关需求  
8. 【功能需求】：系统应允许教师在发布课程时设置先修课程、专业限制或年级要求等准入条件。  
9. 【功能需求】：系统应支持教师管理课程容量，当达到上限时关闭选课通道，并允许开启"候补队列"功能。  
10. 【功能需求】：系统应提供统一的成绩管理界面，支持教师录入各项成绩并计算总评，可导出学生名单。  
11. 【功能需求】：系统应在开学初期标记未确认选课的学生，并向其发送提醒。  
12. 【非功能需求】：成绩录入与导出操作应在合理时间内完成。  

三、教务管理员相关需求  
13. 【功能需求】：系统应提供统一的线上课程替代与学分认定申请入口，学生提交后系统自动关联培养方案。  
14. 【功能需求】：系统应集成全校教室资源数据库，在教师排课时实时校验教室占用情况。  
15. 【功能需求】：系统应支持按多维度生成教学运行报表，包括选课完成率、课程负荷等关键指标。  
16. 【功能需求】：系统应根据各专业培养方案，生成不同年级学生的推荐选课列表。  
17. 【约束性需求】：系统必须符合教育信息系统安全等级保护要求，所有学生隐私数据须加密存储与传输。  
18. 【质量需求】：系统应具有较高的可用性，故障恢复时间较短，并具备操作日志审计功能。
—— 清单结束 ——
```

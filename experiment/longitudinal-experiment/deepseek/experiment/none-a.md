## 推演
```

请你模拟对“本科生”、“任课教师”和“教务管理员”这三类关键用户进行访谈。请分别列出从每类用户那里搜集到的原始反馈。反馈内容应聚焦于他们在实际操作中遇到的具体问题、不便之处、以及他们直接表达的期望。请使用他们可能使用的自然语言来描述，确保反馈是具体、有场景的，而非笼统的概括。

请严格按以下格式输出：
```

```
[问题描述]：选课系统在早上9点开放时，经常点不动或者直接报错，等能进去的时候，热门的通选课早就没名额了。

[问题描述]：想了解一门课的教材信息，得等开课第一周老师说了才知道，有时候想提前预习或者买二手书都来不及。
[具体场景与期望]：我希望在选课系统里，每门课的介绍下面就能提前看到推荐教材清单和ISBN书号，方便我们提前准备。

二、 任课教师主要反馈：
[问题描述]：课程结束后，我需要从好几个不同的Excel表格里手动汇总、核对选课学生名单和最终成绩单，非常容易出错，也很耗时。
[具体场景与期望]：我希望在课程容量满员后，系统能自动关闭选课通道，而不是需要我每天登陆上去手动查看并关闭。

[问题描述]：录入期末成绩时，系统只能一门课一门课地进入，手动输入或粘贴，操作很繁琐，没有批量导入或高效处理的功能。
[具体场景与期望]：我希望能有一个模板，让我在Excel里整理好成绩后，一键导入系统，并且系统能自动进行基本校验（比如成绩是否在0-100分之间）。

[问题描述]：调课或者补课的时候，需要自己一个个通知学生，总有学生说没看到消息，事后又来问。
[具体场景与期望]：我希望在教务系统提交调停课申请并获批后，系统能自动给选课学生发送通知（比如短信或App推送），并让他们确认知悉。

[问题描述]：想了解选我课的学生背景（比如哪些专业的、之前学过哪些先修课），得去找教务员单独查询，很不方便。
[具体场景与期望]：我希望在课程的学生名单页面，能直观看到每个学生的主修专业和已修相关课程情况，方便我因材施教。

[问题描述]：学生的作业和报告通过邮件、网盘链接等各种方式提交，格式乱七八糟，整理、批阅和归档特别麻烦。
[具体场景与期望]：我希望有一个统一的作业提交平台，学生在线提交，系统自动按学号姓名整理好，我可以在线批注、打分并一键反馈给学生。

三、 教务管理员主要反馈：
[问题描述]：目前每学期初都需要人工审核大量的课程替代、学分认定申请，这些申请格式不一，通过邮件或纸质提交，整理和审批流程混乱。
[具体场景与期望]：我们需要系统能根据教学计划，自动生成不同专业、不同年级学生的推荐选课列表，作为他们的参考，这样可以减少大量无效的咨询工作。

[问题描述]：毕业资格审核全靠人工对照培养方案一条条核对，几百个学生审下来眼睛都花了，还容易漏看。
[具体场景与期望]：希望系统能根据每个学生的已修课程和成绩，自动进行毕业预审核，并标记出可能存在问题的学生和缺少的学分类型，我们只需要进行最终复核即可。

[问题描述]：排教室是个老大难问题，既要考虑课程人数、教室容量，还要考虑特殊设备需求，现在全靠Excel和人工经验来回协调，经常排错或冲突。
[具体场景与期望]：我希望系统能提供智能排课功能，我们设定好约束条件（如时间、容量、设备），系统能自动生成一个冲突最少的初步方案供我们调整。

[问题描述]：培养方案或课程信息有调整时，我们发通知邮件，很多老师和学生根本不看，导致后续教学安排出现误解。
[具体场景与期望]：希望重要通知（如方案变更）在系统首页有强提醒弹窗，用户必须阅读确认后才能进行其他操作，并且后台能追踪谁已读谁未读。

[问题描述]：和财务处对接教师课时费计算时，需要手动导出上课名单、课时数等数据，再加工成表格，两边数据经常对不上，来回沟通成本高。
[具体场景与期望]：希望系统能按财务处要求的固定格式，一键生成每学期教师的课时工作量及酬金计算明细表，并确保数据来源权威、统一。
```

## 检验
```
现在对已收集到的原始诉求进行检验。
请从以下两个视角进行评估：
1.  清晰度视角：找出所有表述模糊、存在歧义或过于宽泛的诉求（例如，“系统要好用”、“操作要快”），并将其标记出来。
2.  完整性视角：基于我们已识别的用户角色和核心意图，你认为还有哪些重要的用户角色（如系统运维人员、财务处等）或业务场景（如选修课退补选、跨专业选课）的诉求可能被遗漏了？
3. 需求类型覆盖度视角：检查是否包含功能、非功能、约束性、质量四类需求，若存在类型缺失，需标记为遗漏项。

请直接列出你发现的具体模糊项和可能遗漏的视角以及用户角色。
若没有发现具体模糊项或可能遗漏的视角，且四类需求均覆盖完整，则提示“可以进入交付阶段”
```

**结果**
```
### 评估结果

#### 1. 清晰度视角
经逐条检查，所有原始诉求均描述了具体的问题场景和期望，未发现表述模糊、存在歧义或过于宽泛的表述（如“好用”、“快”等）。诉求均具备可操作性，可直接用于后续需求分析。

#### 2. 完整性视角
**可能遗漏的重要用户角色：**
- **系统运维人员**：关注系统部署、监控、维护、备份、安全、性能优化及故障处理。
- **财务处工作人员**：关注与教务系统数据对接，如课时费计算、学生缴费状态同步等。
- **院系教学负责人/专业负责人**：关注课程设置审批、教学评估数据分析、教师考核等。
- **学生辅导员**：关注学生学业进展监控、预警、选课指导等。
- **图书馆/实验室管理人员**：关注课程资源预约（如图书、实验设备）与教务系统课程安排的联动。
- **跨校交流学生/访学生**：涉及跨校选课、学分互认的特殊流程。

**可能遗漏的关键业务场景：**
- **选修课退补选流程**：学生退课、补选的规则与操作。
- **跨专业选课审批流程**：学生选修其他专业课程时的申请与审核。
- **重修/补考管理**：报名、缴费、成绩录入与覆盖规则。
- **教学评估（学生评教）**：评教问卷发放、回收、统计与反馈。
- **实践环节管理**：实习、毕业设计/论文的选题、过程管理与成绩评定。
- **教材征订与发放**：与课程绑定，统计需求、采购与发放。
- **学业预警机制**：自动识别多门不及格、学分不足等风险并通知学生及辅导员。
- **教师调代课审批流程**：线上申请、审批与记录。
- **教室借用管理**：非课程用途的教室借用申请与调度。
- **国际化课程管理**：全英文课程、外教授课的特殊安排与质量监控。

#### 3. 需求类型覆盖度视角
**已覆盖的类型：**
- **功能需求**：大量覆盖，如冲突检查、成绩推送、自动排课、数据导入导出等。
- **非功能需求**：部分覆盖，涉及性能（系统响应慢）、可用性（操作繁琐）等，但不够系统。

**明显缺失的类型：**
- **约束性需求**：未提及任何技术、业务或合规性限制，例如：
  - 必须与现有校园身份认证系统（如LDAP）集成。
  - 必须符合教育部《教育管理信息化标准》。
  - 数据库必须使用学校指定的Oracle版本。
  - 必须支持主流浏览器（Chrome、Firefox）的最近两个版本。
- **质量需求**：未明确提及系统质量属性，例如：
  - 安全性：数据传输加密、角色权限最小化、防止SQL注入等。
  - 可维护性：代码结构、日志记录、系统文档要求。
  - 可扩展性：未来新增功能模块的便利性。
  - 可靠性：系统可用性达到99.9%，数据备份与恢复机制。

**结论：** 由于存在明显的用户角色、业务场景遗漏，且约束性需求和质量需求缺失，当前需求收集尚未完整，**不能直接进入交付阶段**。建议针对上述遗漏项进行补充访谈和挖掘。
```

```
[问题描述]：数据库备份目前是每周手动全量备份，如果中途系统出问题，可能会丢失几天的数据。恢复演练也很少做，真出事了心里没底。
[具体场景与期望]：我们需要系统支持每日自动增量备份和每周全量备份，并且备份文件能自动同步到异地的存储。最好还能提供一个模拟恢复的测试环境，让我们定期演练。

[问题描述]：经常有老师或学生反馈账号被盗用，或者权限异常。我们现在缺乏有效的安全审计日志，出了问题很难追溯是谁在什么时间做了什么操作。
[具体场景与期望]：所有关键操作（如登录、成绩修改、权限变更）都必须记录详细的日志，包括操作人、时间、IP地址和具体内容。希望能有一个安全的日志查询界面供我们排查问题。

[问题描述]：开发团队有时候直接在生产环境修改数据库来应急，这种做法风险极高，很容易引发数据不一致或更严重的问题。
[具体场景与期望]：必须严格禁止任何绕过系统的后台直接数据库操作。所有的数据变更都必须通过系统的正规业务界面或经过严格审批的维护流程来完成，并留有完整记录。

## 五、财务处工作人员主要反馈
[问题描述]：计算教师课时费时，需要从教务系统手动导出课程、学生选课名单和课时数，再手工计算，效率低且易出错。
[具体场景与期望]：希望教务系统能提供标准化的课时费计算报表，自动关联课程、教师、选课人数和课时，并按照学校规定的课时费标准自动计算，生成可直接用于发放的表格。

[问题描述]：学生缴费状态（如学费、住宿费）与选课、考试资格挂钩，但目前缴费系统与教务系统不互通，需要人工比对，经常出现学生欠费但仍能选课的情况。
[具体场景与期望]：希望教务系统能实时获取学生的缴费状态，并在学生选课、考试报名等环节自动进行拦截，提示学生先缴费。

[问题描述]：奖助学金发放需要核实学生的在校状态和学业表现，目前需要找教务处和各个学院分别要数据，流程很长。
[具体场景与期望]：希望教务系统能提供标准化的学生学籍和成绩数据接口，让我们的奖助学金系统能自动获取相关信息，减少人工核对。

## 六、院系教学负责人主要反馈
[问题描述]：每学期排课时，需要协调各专业的课程安排，避免时间冲突，目前靠人工沟通，效率低且容易出错。
[具体场景与期望]：希望系统能提供院系级别的课程排课冲突检查，并能在调整时实时提示冲突，协助我们高效安排课程。

[问题描述]：需要对教师的教学质量进行评估，但目前学生评教数据分散，难以汇总分析。
[具体场景与期望]：希望系统能自动汇总每门课的学生评教结果，并生成院系、教师、课程等不同维度的统计分析报告，支持导出。

[问题描述]：制定和修改培养方案时，需要参考历年的执行情况和学生反馈，但现在数据分散，难以做出科学决策。
[具体场景与期望]：希望系统能提供培养方案执行情况的数据分析，包括每门课程的选课人数、成绩分布、学生评价等，帮助我们优化课程设置。

## 七、学生辅导员主要反馈
[问题描述]：很难及时掌握所带学生的学业情况，比如挂科、学分不足等，通常等到问题严重了才发现。
[具体场景与期望]：希望系统能自动监控学生的学业进展，当出现挂科、学分不足等情况时，自动给我发送预警信息，以便及时干预。

[问题描述]：每学期都要手动统计学生的选课情况，了解他们的学习负担，现在只能一个个学生去问。
[具体场景与期望]：希望系统能提供我所带班级的选课情况汇总，包括每个学生选了哪些课、学分总数，并标记出选课过多或过少的学生。

[问题描述]：学生来找我咨询职业规划或考研时，我需要先了解他们的成绩单，但每次都要让学生自己去打印，很不方便。
[具体场景与期望]：希望系统能授权我查看所带学生的完整成绩单和学习进度，以便提供更有针对性的指导。

## 八、约束性需求
1. 系统必须与学校现有的统一身份认证系统（如LDAP）集成，用户使用同一账号密码登录。
2. 系统必须符合教育部《教育管理信息化标准》和学校的数据安全管理办法。
3. 系统数据库必须使用学校指定的Oracle 19c版本，并部署在学校私有云平台上。
4. 系统必须支持主流浏览器（Chrome、Firefox、Edge）的最近两个稳定版本。
5. 系统必须提供标准的API接口，以便与学校财务系统、图书馆系统等第三方系统对接。
6. 所有敏感数据（如成绩、个人信息）在传输和存储时必须加密。
7. 系统必须支持至少5000名用户同时在线进行选课操作。
8. 所有用户界面必须支持中文，关键功能界面应提供英文版本以满足国际化需求。

## 九、质量需求
1. 性能：选课、成绩查询等核心功能在高峰时段（如选课开放时）的响应时间不超过3秒。
2. 可用性：系统核心功能（选课、成绩录入）的可用性不低于99.5%。
3. 安全性：所有数据传输必须采用HTTPS加密；用户密码必须加密存储；系统应具备防SQL注入、XSS攻击等能力。
4. 可维护性：系统应提供完整的操作日志，关键业务操作（如成绩修改、权限变更）必须记录操作人、时间、IP和具体内容。
5. 可扩展性：系统应采用模块化设计，未来新增功能模块（如实习管理）时，不影响现有核心功能。
6. 可靠性：系统应具备容错能力，单点故障不应导致整个系统瘫痪；数据丢失恢复时间不超过2小时。
7. 易用性：系统主要功能的操作学习成本应低于15分钟；界面设计应符合主流操作习惯。
8. 兼容性：系统应能在Windows、macOS、iOS和Android系统的主流版本上正常运行。
```

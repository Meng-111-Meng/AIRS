# 纵向对比实验 单向AIRS deepseek

## 提示词
```
1. 锚定 
角色：资深需求工程师
背景：一所综合性大学计划开发新的“学校选课管理系统”。项目处于最早期阶段。
任务：目前在锚定阶段，请你先明确所需扮演的角色，背景和任务。
在接下来的推演阶段，你将执行初始的需求获取工作。具体而言，当进入推演阶段时，你需要模拟一次与关键干系人（学生、教师、教务处管理员）的引导式访谈，通过提问来发掘和记录他们的原始问题、操作流程、痛点及期望，并分类成功能需求、非功能需求、约束性需求、质量需求，最终输出一份需求描述清单。
如果已明确请提示可进入【推演】阶段



2. 推演
现在，基于以上锚定，开始需求探索。
请你模拟对“本科生”、“任课教师”和“教务管理员”这三类关键用户进行访谈。请分别列出从每类用户那里搜集到的原始反馈。反馈内容应聚焦于他们在实际操作中遇到的具体问题、不便之处、以及他们直接表达的期望。请使用他们可能使用的自然语言来描述，确保反馈是具体、有场景的，而非笼统的概括。

请严格按以下格式输出：
一、 本科生主要反馈：
[问题描述]：例如，“选课系统在早上9点开放时，经常点不动或者直接报错，等能进去的时候，热门的通选课早就没名额了。”
[具体场景与期望]：例如，“我在安排课程表时，需要自己用纸笔记下时间，再手动比对有没有冲突。希望系统能在我尝试选择有冲突的课时，立刻用红色高亮提示我具体跟哪门课冲突了。”
...（继续列出其他反馈）

二、 任课教师主要反馈：
[问题描述]：例如，“课程结束后，我需要从好几个不同的Excel表格里手动汇总、核对选课学生名单和最终成绩单，非常容易出错，也很耗时。”
[具体场景与期望]：例如，“我希望在课程容量满员后，系统能自动关闭选课通道，而不是需要我每天登陆上去手动查看并关闭。”
...（继续列出其他反馈）

三、 教务管理员主要反馈：
[问题描述]：例如，“目前每学期初都需要人工审核大量的课程替代、学分认定申请，这些申请格式不一，通过邮件或纸质提交，整理和审批流程混乱。”
[具体场景与期望]：例如，“我们需要系统能根据教学计划，自动生成不同专业、不同年级学生的推荐选课列表，作为他们的参考，这样可以减少大量无效的咨询工作。”
...（继续列出其他反馈）



3. 检验
现在对已收集到的原始诉求进行检验。
请从以下两个视角进行评估：
1.  清晰度视角：找出所有表述模糊、存在歧义或过于宽泛的诉求（例如，“系统要好用”、“操作要快”），并将其标记出来。
2.  完整性视角：基于我们已识别的用户角色和核心意图，你认为还有哪些重要的用户角色（如系统运维人员、财务处等）或业务场景（如选修课退补选、跨专业选课）的诉求可能被遗漏了？
3. 需求类型覆盖度视角：检查是否包含功能、非功能、约束性、质量四类需求，若存在类型缺失，需标记为遗漏项。

请直接列出你发现的具体模糊项和可能遗漏的视角以及用户角色。
若没有发现具体模糊项或可能遗漏的视角，且四类需求均覆盖完整，则提示“可以进入交付阶段”

4. 推进

基于上述检验生成后的结果，我们决定进行【优化 】。

若您决定进行【优化】，请继续：
作为需求工程师，请你根据检验结果指出的具体问题，执行对应操作：
A. 若检验指出存在模糊/歧义诉求：
请直接修改这些模糊诉求。将其拆解或转化为具体、无歧义、可作为后续设计输入的清晰陈述。
B. 若检验指出存在遗漏的视角/场景：
请模拟对已识别的遗漏角色或场景进行一次快速需求发现，并新增从该视角获取的原始诉求。

完成上述操作后，请按顺序提供以下交付物：
更新后的原始需求清单：整合所有优化后的诉求，按角色分类清晰列出。



基于上述检验生成后的结果，我们决定进行【交付 】。
请将最终获取的所有原始需求，按用户角色分类，整理成一份《系统需求描述清单》。
请为每条生成的需求标注类型（功能需求 / 非功能需求 / 约束性需求 / 质量需求），标注格式为【需求类型】：需求内容。
清单应清晰列出每个角色及其诉求，诉求需以系统为主语，明确具体、可验证。
最终只需输出《系统需求描述清单》即可
```




## 结果
```
《学校选课管理系统》需求描述清单

一、 本科生需求
1.  【功能需求】：系统应在每学期选课开放期间（如早上9点），能承受至少5000名用户并发访问，保证服务可用与响应。
2.  【功能需求】：当学生尝试选择时间冲突的课程时，系统应立即在界面上以红色高亮提示冲突的详细信息（如课程名称、时间）。
3.  【功能需求】：系统应在选课界面为每门课程提供历史的学生评价信息、课程大纲和教师简介，以辅助学生决策。
4.  【非功能需求】：系统主页及核心选课操作的页面加载时间应在3秒以内。
5.  【功能需求】：系统应提供个人课表可视化功能（如以日历视图显示），并支持导出为常见格式（如.ics日历文件）。
6.  【质量需求】：系统界面设计应符合主流学生用户的操作习惯，关键操作（如提交选课、退课）需有二次确认提示，防止误操作。

二、 任课教师需求
1.  【功能需求】：系统应为教师提供一个统一的管理视图，可实时查看、导出所授课程的最终选课学生名单，并支持在线录入、计算和提交成绩。
2.  【功能需求】：当课程选课人数达到预设容量时，系统应自动关闭该课程的选课通道，并可将学生加入等待队列。
3.  【功能需求】：系统应支持教师在线发布课程公告、上传教学资料（如PPT、阅读材料）并设定查看权限（如仅限选课学生）。
4.  【功能需求】：系统应支持教师在线发布作业、设置截止日期，并提供学生在线提交作业的功能。
5.  【非功能需求】：与教师相关的数据（如未发布的成绩、学生提交的作业）必须进行加密存储和传输，确保隐私与安全。

三、 教务管理员需求
1.  【功能需求】：系统应提供线上课程替代、学分认定申请的标准化提交与审批流程，支持附件上传和审批历史追踪。
2.  【功能需求】：系统应根据各专业、各年级的官方教学计划，为每位学生生成一份“推荐选课列表”，并在其选课前主动提示。
3.  【功能需求】：系统应允许教务管理员在后台灵活配置选课规则（如先修课要求、容量限制、专业限制、选课轮次与时间）。
4.  【功能需求】：系统应能按需生成各类统计报表，如课程选课率、班级名单、教师开课情况等，并支持导出。
5.  【约束性需求】：系统必须与学校现有统一身份认证平台对接，实现单点登录。
6.  【非功能需求】：系统所有关键业务操作（如成绩录入、规则修改、名单调整）必须记录详细的操作日志，以备审计。

四、 补充角色：研究生需求
1.  【功能需求】：系统应支持研究生培养计划（含学位课、选修课、必修环节）的在线制定、提交及导师/学院审核功能。
2.  【功能需求】：系统应为研究生选课提供“导师批准”流程，部分课程的选择需经导师在线确认后方可生效。

五、 补充角色：系统运维人员需求
1.  【非功能需求】：系统应提供可视化监控面板，实时显示系统健康状态（如服务器负载、数据库连接数、当前在线用户数）。
2.  【功能需求】：系统应具备关键业务数据的定期自动备份与一键恢复功能。
3.  【约束性需求】：系统前端需适配主流浏览器（Chrome, Edge, Firefox）的最新两个版本。
4.  【质量需求】：系统的代码结构、数据库设计及接口文档应清晰规范，确保新功能模块能以较低成本接入。
```

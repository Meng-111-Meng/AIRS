```
一、 本科生主要反馈：  
[问题描述]：选课系统在早上9点开放时，经常点不动或者直接报错，等能进去的时候，热门的通选课早就没名额了。

[问题描述]：退课和补选的时间窗口太短，而且通知不明显，去年我就因为错过一天的补选时间，导致整个学期少修了一门学分。  
[具体场景与期望]：能不能在个人主页上有个清晰的日程提醒模块，比如“距离退课截止还有2天”，并支持微信或短信推送？  

[问题描述]：查不到课程的教学大纲和往年的评分情况，只能靠学长学姐口口相传，选到“水课”或“地狱难度”课全靠运气。  
[具体场景与期望]：希望每门课页面能展示教师上传的教学大纲、考核方式，甚至匿名的学生历史评价（像选餐厅看评分那样）。  

二、 任课教师主要反馈：  
[问题描述]：课程结束后，我需要从好几个不同的Excel表格里手动汇总、核对选课学生名单和最终成绩单，非常容易出错，也很耗时。  
[具体场景与期望]：我希望系统能一键导出当前学期所有选课学生的完整名单（含学号、专业、班级），并支持在线录入平时分、期中、期末成绩，自动计算总评。  

[问题描述]：经常有学生在课程开始两周后还试图加课，理由五花八门，但我根本不知道他们是否符合旁听或特殊加课条件。  
[具体场景与期望]：系统应该只允许在规定时间内选课，超期后必须由教务处审批才能加入，并自动通知我该学生是否获得批准，而不是让我私下判断。  

[问题描述]：我想限制选课学生的先修课程要求，比如“数据库原理”必须先修过“程序设计基础”，但现在的系统没法设置这种规则。  
[具体场景与期望]：在发布课程时，能勾选必要的先修课程，系统自动拦截不符合条件的学生选课申请，并给出明确提示。  

[问题描述]：教室容量和实际选课人数对不上，有时系统显示50人上限，结果来了70人，后排学生站着听课，教学效果很差。  
[具体场景与期望]：系统应根据教室真实容量自动设定课程最大人数，且不允许超限选课，避免现场混乱。  

三、 教务管理员主要反馈：  
[问题描述]：目前每学期初都需要人工审核大量的课程替代、学分认定申请，这些申请格式不一，通过邮件或纸质提交，整理和审批流程混乱。  
[具体场景与期望]：我们需要一个在线申请-审批工作流，学生在系统里填写结构化表单，自动关联其已修课程和培养方案，审批状态全程可追踪。  

[问题描述]：排课时经常出现教室资源冲突，比如同一间教室被两个学院同时安排了课程，协调起来非常被动。  
[具体场景与期望]：系统应集成全校教室资源池，在教师提交开课计划时实时校验时间+地点是否已被占用，冲突时禁止提交或强制调整。  

[问题描述]：每学期末要手动统计各专业选课完成率、未达标学生名单，工作量巨大，还容易漏人。  
[具体场景与期望]：系统能按专业、年级自动生成学业进度报告，标红未完成核心课程或学分不足的学生，并支持一键导出预警名单。  

[问题描述]：我们制定的教学计划和实际学生选课行为脱节，比如某门必修课明明安排在大三，但很多学生拖到大四才选，影响毕业审核。  
[具体场景与期望]：系统应基于学生所属专业的培养方案，动态生成“推荐选课路径”，并在学生偏离路径时弹出警示（如“您尚未修读XX必修课，建议本学期优先选择”）。

## 检验
```
现在对已收集到的原始诉求进行检验。
请从以下两个视角进行评估：
1.  清晰度视角：找出所有表述模糊、存在歧义或过于宽泛的诉求（例如，“系统要好用”、“操作要快”），并将其标记出来。
2.  完整性视角：基于我们已识别的用户角色和核心意图，你认为还有哪些重要的用户角色（如系统运维人员、财务处等）或业务场景（如选修课退补选、跨专业选课）的诉求可能被遗漏了？
3. 需求类型覆盖度视角：检查是否包含功能、非功能、约束性、质量四类需求，若存在类型缺失，需标记为遗漏项。

请直接列出你发现的具体模糊项和可能遗漏的视角以及用户角色。
若没有发现具体模糊项或可能遗漏的视角，且四类需求均覆盖完整，则提示“可以进入交付阶段”
```

**结果**
```
经对已收集的原始诉求进行系统性检验，评估结果如下：

1. 清晰度视角：模糊或宽泛表述识别

在当前三类用户的原始反馈中，所有诉求均以具体场景、行为或期望形式表达，未出现明显模糊、歧义或过于宽泛的表述（如“系统要稳定”“界面要美观”等）。  
例如：
- “选课系统在早上9点开放时，经常点不动或者直接报错” → 明确时间、现象、后果；
- “希望系统能实时更新余量” → 明确功能目标与用户痛点；
- “系统应根据教室真实容量自动设定课程最大人数” → 具体规则与业务逻辑清晰。

✅ 结论：无模糊项需修正。

2. 完整性视角：潜在遗漏的用户角色与业务场景

尽管学生、教师、教务管理员覆盖了选课流程的核心三方，但以下重要角色和场景仍可能被遗漏：

（1）遗漏的用户角色：
- 系统运维/IT支持人员：负责系统部署、监控、故障响应、数据备份等。其诉求涉及高可用性、日志审计、灾备机制等，目前未体现。
- 院系教学秘书：介于教师与教务处之间，常负责本院课程审核、开课计划初审、学生选课异常处理等，其工作流未被覆盖。
- 跨校/交换生管理人员：若学校有联合培养或交换项目，其学分转换、课程匹配需求可能特殊。
- 财务处（间接相关）：部分高校按学分收费，选课结果直接影响缴费金额，退补选可能触发退费流程。

（2）遗漏的关键业务场景：
- 跨专业/辅修/双学位选课流程：当前反馈聚焦主修课程，但辅修学生常面临权限、先修要求、名额优先级等问题。
- 候补队列（Waitlist）机制：多名学生提到“名额瞬间消失”，但未提出“排队候补”诉求，而这是现代选课系统常见功能。
- 课程冲突的智能调课建议：仅提到冲突提示，未涉及系统是否可推荐替代时段或相似课程。
- 国际化支持：如多语言界面（针对留学生）、时区适配（针对在线课程）等，在综合性大学中可能必要。
- 数据导出与对接第三方系统：如与毕业审核系统、成绩系统、一卡通系统等的接口需求未被提及。

✅ 结论：存在完整性缺口，尤其在运维角色、跨专业选课、候补机制等方面。

3. 需求类型覆盖度视角

对原始反馈进行归类分析：

- 功能需求：大量存在（如冲突检测、先修课校验、自动排课校验、成绩录入、审批流等）→ ✅ 覆盖充分。
- 非功能需求：
  - 性能：“选课高峰系统卡顿” → 隐含高并发性能要求；
  - 可用性：“希望有微信提醒” → 涉及通知渠道多样性；
  - 安全性：虽未明说，但“学生成绩”“选课权限”等隐含访问控制需求；
  - 可靠性：“数据不同步” → 隐含数据一致性要求。
  → ✅ 非功能需求已有初步体现。
- 约束性需求：
  - “必须先修某课程才能选” → 业务规则约束；
  - “教室容量限制” → 物理资源约束；
  - “退补选时间窗口” → 时间策略约束。
  → ✅ 存在明确约束。
- 质量需求（常与非功能重叠，但更强调用户体验与系统品质）：
  - 实时性（名额更新）、准确性（冲突判断）、易用性（可视化课表）、可追溯性（审批记录）等均有体现。
  → ✅ 基本覆盖。

⚠️ 但注意：上述非功能与质量需求多为隐含，尚未显式提炼为可验证指标（如“系统需支持5000人同时在线选课，响应时间<2秒”）。不过，在原始用户反馈阶段，此属正常。

✅ 结论：四类需求均有覆盖，无类型缺失。

综合判断：

虽然需求类型覆盖完整且原始表述清晰，但完整性视角存在明显缺口——特别是系统运维人员、教学秘书等关键干系人未纳入访谈，且候补队列、跨专业选课、系统集成等重要场景缺失。

因此，暂不能进入交付阶段。

建议行动：
在进入正式需求规格说明书编写前，应：
1. 补充对系统运维团队和院系教学秘书的访谈；
2. 主动探查跨专业选课、候补机制、系统对接等边缘但关键场景；
3. 明确是否支持国际化、多终端（如移动端）访问等扩展性需求。

待补充后，方可进入交付阶段。
```

```
根据检验结果，当前原始诉求无模糊或歧义项需修改（A项无需操作），但存在完整性缺口（B项需补充）。现执行【优化】操作，重点补充遗漏角色与场景。

- 新增业务场景：跨专业/辅修选课、候补队列机制、系统集成对接

补充原始反馈（按角色）

四、 院系教学秘书主要反馈：
[问题描述]：每学期初我们要手动核对教师提交的开课计划是否符合培养方案，比如课程代码是否正确、学分是否匹配，经常发现错误时已经临近选课，改起来很被动。  
[具体场景与期望]：希望系统在教师提交开课申请时，自动比对其所属专业的培养方案，若课程属性（如学分、课程类型）不一致，直接标红并阻止提交，直到修正为止。  

[问题描述]：经常有学生来找我处理“明明满足先修条件却无法选课”的问题，查了半天才发现是系统没同步上学期刚通过的成绩。  
[具体场景与期望]：选课资格判断应基于最新成绩库（包括刚录入但未正式发布的成绩），或者至少允许教学秘书临时授权特定学生加课，并留痕备案。  

[问题描述]：辅修学生选我们学院的课时，和主修生混在一起抢名额，导致辅修生根本选不上必修课。  
[具体场景与期望]：希望开课时能设置“主修保留名额”和“辅修开放名额”，比如一门课50人，40个给主修，10个专供辅修，系统自动分流。  

五、 系统运维工程师主要反馈：
[问题描述]：每年选课高峰期，服务器CPU和数据库连接池都会被打满，虽然提前扩容，但还是会出现502错误，而且日志分散在多个节点，排查慢。  
[具体场景与期望]：系统需要内置性能监控面板，实时显示并发用户数、响应延迟、数据库负载；同时支持自动限流和排队机制，而不是直接崩溃。  

[问题描述]：每次教务处临时调整选课时间（比如因节假日顺延），我们都得手动改配置文件、重启服务，容易出错。  
[具体场景与期望]：所有业务规则（如选课开始/结束时间、退课窗口）都应该能在管理后台动态配置，无需代码发布或重启系统。  

[问题描述]：去年发生过一次数据误删，因为没有操作审计日志，不知道是谁删的，恢复也花了半天。  
[具体场景与期望]：所有关键操作（如删除课程、强制退选、修改学分）必须记录操作人、时间、IP和变更前后值，并支持按条件检索。  

六、 跨专业/辅修学生（代表性场景补充）：

【FR】系统应按专业、年级自动生成学业进度报告，标红未完成核心必修课或总学分不足的学生，并支持一键导出预警名单。  
【FR】系统应基于学生所属专业的培养方案，动态生成个性化“推荐选课路径”，并在学生偏离路径时弹出警示提示。  
【CR】所有课程的开课信息（课程代码、学分、类型）必须与官方培养方案一致，否则不得进入选课池。  
【NFR】系统应支持每学期处理不少于10万条选课记录，数据一致性误差率低于0.001%。  
【QR】审批流程应支持多级审核、意见填写与历史版本回溯，确保过程透明可审计。

四、院系教学秘书

【FR】系统应在教师提交开课申请时，自动比对其所属专业的培养方案，若课程属性（学分、类型、代码）不匹配，则标红并阻止提交。

【FR】系统应支持在开设课程时分别设置“主修保留名额”和“辅修/跨专业开放名额”，并自动分流选课请求。  
【CR】辅修课程的选课权限应严格限定于已注册该辅修项目的学生。  
【QR】异常处理操作（如强制加课）应提供简洁的表单界面，5分钟内可完成单次授权。

五、系统运维工程师

【FR】系统应内置实时性能监控面板，显示当前并发用户数、平均响应延迟、数据库连接池使用率及错误率。  
【FR】系统应支持在高并发场景下自动启用排队机制（如虚拟等待队列），而非直接返回502错误。  
【FR】所有关键业务规则（如选课开始/结束时间、退课窗口、补选周期）应可通过管理后台动态配置，无需重启服务或部署代码。  
【FR】系统应对所有关键操作（如删除课程、强制退选、修改学分、调整名额）记录操作人、操作时间、IP地址及变更前后的数据值。  
【NFR】系统应支持7×24小时运行，年度计划外停机时间不超过4小时。  
【NFR】操作日志应集中存储，支持按操作人、时间范围、操作类型等条件检索，响应时间不超过3秒。  
【CR】系统必须部署在符合等保二级要求的校园私有云环境中，禁止敏感数据外传。  
【QR】故障发生时，系统应提供明确的错误码与自助排查指引，便于快速定位问题。

六、跨专业/辅修学生（代表性场景）

【FR】系统应支持课程可见性按“学科门类”“学院”或“自定义标签”（如“面向全校开放”）进行设置，而非仅限于专业硬性过滤。

【CR】跨专业选课若涉及先修要求，系统应允许院系设置“等效课程”映射关系（如“高等数学A”等效于“微积分I”）。
